# Claude Assistant Instructions

## 一、用户与项目背景

### 1. 用户背景

- 本科专业：信息管理与信息系统（东华大学旭日工商管理学院）
- 项目用途：本科毕业设计
- 当前阶段目标：
  - 近期有毕设阶段检查，需向导师演示可运行的系统
  - 完成一个「可运行、可演示、可讲解」的 RAG 系统
  - 在阶段检查中展示系统核心功能与技术实现思路

### 2. 项目简介

本项目是一个**基于检索增强生成（RAG）的个性化食谱推荐系统**。

系统以结构化食谱数据构建领域知识库，通过向量检索将外部知识引入大模型生成过程，使模型在推荐结果中具备：

- 更强的领域相关性
- 可追溯的知识来源
- 对饮食约束条件（食材禁忌、营养目标、菜系偏好）的更好适配能力

项目重点不在于"推荐算法性能对比"，而在于**验证 RAG 在垂直领域知识增强中的工程可行性与解释性优势**。

### 3. 毕设选题范式

本项目属于**信息系统开发类**毕业设计，需满足以下要求：

- 以特定场景（饮食推荐）为导向，开展信息系统的设计与开发
- 包含完整流程：需求分析 → 系统设计 → 系统实现 → 测试验证
- 体现信息管理与信息系统专业的核心能力：信息采集、存储、处理、展示
- 突出RAG技术在垂直领域的应用价值，而非算法研究

### 4. 数据源

- 主食谱数据：下厨房结构化食谱数据（`recipe_corpus_full.json`）
- 主营养数据：**中国食物成分表（Excel）**（离线导入、本地存储）
- 补充营养数据：**FatSecret API**（仅在本地缺失食材时按需调用）
- 数据处理方式：先对中国食物成分表进行标准化预处理并构建本地营养库，再将食谱主数据与营养字段对齐；仅对缺失项调用 FatSecret 并回写本地缓存

## 二、项目定位与边界

### 1. 项目类型

- 工程型大模型应用
- 非算法研究项目
- 不涉及模型训练或参数微调

### 2. 系统架构

采用**前后端分离**架构：

| 层级       | 技术选型       | 职责                                 |
| ---------- | -------------- | ------------------------------------ |
| 前端       | **Vue3**       | 用户约束输入、推荐结果展示、偏好管理 |
| 后端       | **Flask**      | 提供推荐接口、数据接口、偏好管理接口 |
| RAG 引擎   | **LlamaIndex** | 文档索引、向量检索、LLM 生成         |
| 向量数据库 | **Qdrant**     | 向量索引存储与高效相似度检索         |

### 3. 当前阶段范围

- 提供完整的前后端交互系统（非单一演示界面）
- 后端提供清晰的 RESTful API 接口
- 前端 Vue3 应用支持：
  - 用户自然语言输入
  - 饮食约束选择（食材禁忌、营养目标、菜系偏好）
  - 推荐结果展示（含推荐解释、检索来源）
- 支持检索来源展示（可追溯性）

## 三、技术约束与实现原则

### 1. 技术栈选型

| 组件           | 技术选型                         | 说明                                           |
| -------------- | -------------------------------- | ---------------------------------------------- |
| RAG 框架       | **LlamaIndex**                   | 主框架，负责文档索引、检索、生成 Pipeline 编排 |
| 向量数据库     | **Qdrant**（本地持久化模式）     | 支持向量检索与过滤，和 LlamaIndex 集成良好     |
| Embedding 模型 | **Qwen3-Embedding-0.6B**（本地） | 中文能力强，轻量级，与 LLM 同系列              |
| LLM            | **通义千问 Qwen API**            | 免费额度充足，中文能力强，国内直接访问         |
| 主营养数据源   | **中国食物成分表（Excel）**      | 作为营养字段主来源，离线清洗后本地查询         |
| 营养补全接口   | **FatSecret API**                | 仅用于缺失食材补全，控制月度调用额度           |
| 后端框架       | **Flask**                        | 轻量级 Python Web 框架，提供推荐接口、数据接口 |
| 前端框架       | **Vue3**                         | 现代前端框架，构建面向用户的交互界面           |

### 2. RAG 实现原则

- 数据来源：下厨房食谱语料（`recipe_corpus_full.json`）+ 中国食物成分表（主）+ FatSecret API（补）
- 文档构建方式：通过字段拼接规则，将结构化数据转为可检索文本
- 使用 LlamaIndex 管理文档索引与检索流程
- 使用 Qdrant 持久化向量索引，并结合营养字段进行检索过滤
- 营养处理策略：采用“预处理优先、在线补全兜底”模式；优先查询本地营养库，仅在缺失时调用 FatSecret 并回写缓存
- RAG 流程需具备清晰的步骤划分，便于毕设答辩讲解

### 3. 框架使用策略

- 使用 LlamaIndex 作为主框架，利用其成熟的 RAG Pipeline 能力
- 核心流程（文档处理、向量化、检索、上下文拼接）需理解原理，允许使用框架封装
- 避免使用过度抽象的 Agent / Workflow 自动化流程

## 五、协作规则

1. 在实现关键模块前，需先说明设计思路
2. 若存在多种可选方案，请先列出并说明优缺点
3. 禁止在未说明理由的情况下引入新的技术栈
4. 所有代码需满足：
   - 可运行
   - 结构清晰
   - 能被我理解并讲解核心逻辑（框架内部代码无需逐行解释）

6. 全程使用中文与我交流，包括思考链

在未明确说明前，请不要扩展 Agent、多模型调度等高级功能。
