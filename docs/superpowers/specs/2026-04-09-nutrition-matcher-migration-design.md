# NutritionMatcher 适配新营养数据库设计文档

## 1. 背景与目标

### 1.1 背景

当前 `NutritionMatcher`（`src/data/nutrition_matcher.py`，约 1056 行）设计用于匹配旧的 `food.db`（来源：不完整的中国食物成分表 Excel）。该数据库**缺失大量食物类别**（糖类、油脂、坚果、调味料等），导致批量匹配率很低（唯一匹配率仅 17.6%）。

用户找到了更完整的营养数据库 `food_nutrition.db`（来源：中国疾病预防控制中心营养与健康所 & 中国营养学会，1346 条记录），覆盖了所有之前缺失的类别。

### 1.2 目标

将 `NutritionMatcher` 从旧 `food.db` 迁移到新 `food_nutrition.db`，具体包括：

1. **重写数据库读取层**：适配新数据库的表结构和带单位文本格式
2. **精简 COMMON_SEASONINGS**：只保留真正零营养的条目
3. **重建 ALIAS_MAP**：匹配新数据库的命名格式
4. **验证匹配率提升**：目标 ≥ 50% 唯一匹配率（从 17.6% 提升）

### 1.3 设计决策记录

| 决策 | 选项 | 结论 | 理由 |
|------|------|------|------|
| 数据源 | 合并两个DB / 只用新DB | **只用新DB** | 旧DB数据不够准确 |
| 能量单位 | kJ / kcal / 两者都存 | **kcal** | 用户更习惯千卡 |
| 中间数据库 | 预处理到中间DB / 直接读取 | **直接读取** | 减少数据流转 |
| 实现方式 | 原地改造 / 拆分重构 | **原地改造** | 先验证效果，后续再拆分 |
| 未匹配食材 | 设 (0,0,0,0) / 标记未匹配 | **标记未匹配** | 不编造数据，用户人工补充 |
| 营养字段 | 四大 / 加膳食纤维 / 更多 | **四大营养素** | 保持现状 |

## 2. 新数据库分析

### 2.1 结构

- **表**：`j_food_nutrition`（1346 行）+ `j_food_categories`（111 行）
- **主要列**：`id`, `cate_id`, `food_name`, `alias_name`, `energy`, `protein`, `fat`, `carbohydrate`（共 36 列）
- **关联**：`j_food_nutrition.cate_id` → `j_food_categories.cate_id`

### 2.2 数据格式特征

**带单位文本值**：所有营养数值以文本存储，带单位后缀：

```
"1698kJ"    → 1698.0 (kJ)
"0.1g"      → 0.1 (g)
"5.40μg"    → 5.4 (μg)
```

**特殊值**：
- `"—"` = 未测定 → 视为 0
- `"Tr"` = 未检出（微量） → 视为 0
- `""` = 空 → 视为 0

**食物名称格式**（括号使用不一致）：

```
小麦粉(标准粉)           ← 半角括号(描述)
糖（白砂糖）             ← 全角括号（描述）
油（橄榄油)              ← 混合括号
大蒜[蒜头](鲜)           ← [别名](状态)
辣椒酱[辣椒糊]           ← [别名]
猪肉(奶面)[硬五花]       ← (描述)[别名]
蛋（鸡蛋，均值)          ← 混合括号
```

**`alias_name` 字段**：仅 76 条有值，大多为品牌商品名（如 `"商品名：100％纯牛奶"`），价值有限，不作为匹配来源。

### 2.3 调味料类别覆盖

新数据库覆盖的调味料类别（旧DB完全缺失的）：

| 类别 | 数量 | 示例 |
|------|------|------|
| 糖 | 4 | 白砂糖、绵白糖、冰糖、红糖 |
| 蜂蜜 | 5 | 蜂蜜及变体 |
| 植物油 | 23 | 菜籽油、花生油、芝麻油、橄榄油等 |
| 动物油脂 | 3 | 牛油、羊油、猪油 |
| 酱油 | 4 | 酱油均值、黄豆酱油、老抽、生抽 |
| 醋 | 3 | 醋均值、白醋、陈醋 |
| 酱 | 23 | 豆瓣酱、花生酱、甜面酱、芝麻酱等 |
| 辛香料 | 4 | 胡椒粉、五香粉、咖喱粉、花椒 |
| 盐味精等 | 6 | 精盐、味精、鸡精、酵母、料酒 |

## 3. 详细设计

### 3.1 `_parse_value()` — 新增辅助方法

解析带单位的营养值文本，返回浮点数：

```python
def _parse_value(self, raw: str) -> float:
    """解析带单位的营养值文本
    
    '1698kJ' → 1698.0 | '0.1g' → 0.1 | '—'/'Tr'/'' → 0.0
    """
    if not raw or raw.strip() in ('—', 'Tr', '─', '…'):
        return 0.0
    return float(re.sub(r'[a-zA-Zμ%]+$', '', raw.strip()))
```

### 3.2 `_load_database()` — 重写

**输入**：`food_nutrition.db` 的 `j_food_nutrition` 表

**处理流程**：
1. 查询 `SELECT id, food_name, energy, protein, fat, carbohydrate FROM j_food_nutrition`
2. 对每条记录：
   - 解析 `energy`（kJ → kcal：`÷ 4.184`，保留一位小数）
   - 解析 `protein`、`fat`、`carbohydrate`（去单位 `g`）
   - 存入 `_food_map`：`food_name → (id, food_name, energy_kcal, protein, fat, carbs)`
3. 构建别名索引 `_alias_to_main`：
   - 从 `food_name` 中的 `[别名]` 提取（如 `辣椒酱[辣椒糊]` → `辣椒糊` → 原名）
   - 从 `food_name` 中提取**纯名**（去掉所有类型括号）：如 `大蒜[蒜头](鲜)` → `大蒜`
   - 括号统一处理（兼容全角/半角混用）
4. 构建 `_food_names` 列表（用于模糊匹配）

**`NutritionInfo.energy` 类型变更**：`int` → `float`（kJ→kcal 转换产生小数）

### 3.5 匹配流程（不变）

```
1. COMMON_SEASONINGS 检查 → 命中返回 (0,0,0,0)
2. ALIAS_MAP 检查 → 映射到DB名称，查 _food_map
3. DB 精确匹配 → 查 _food_map
4. DB 别名匹配 → 查 _alias_to_main → _food_map
5. DB 包含匹配 → 遍历 _food_map，去括号后比较
6. 模糊匹配 → rapidfuzz，阈值 ≥60
7. 未匹配 → 返回 None
```
