## 一、数据清洗模块设计

### 1.1 需求分析

原始数据（155万条食谱）存在以下问题：

- 可能存在完全重复的食谱
- recipeInstructions为空（无法使用）
- dish字段中19.87%（约31万条）标记为"Unknown"
- description字段27.95%为空
- 部分dish字段提取不正确
- 可能存在错别字

### 1.2 清洗策略与优先级

| 优先级         | 清洗任务                   | 实现方式                          | 理由                         |
| -------------- | -------------------------- | --------------------------------- | ---------------------------- |
| **P0（必须）** | 删除recipeInstructions为空 | 直接过滤 `len(instructions) == 0` | 无步骤无法推荐，影响RAG质量  |
| **P0（必须）** | 删除完全重复               | MD5哈希去重                       | 避免索引冗余，提升效率       |
| **P1（重要）** | 填充空字段                 | description→"无"                  | 确保文档完整性，避免None错误 |
| **P3（跳过）** | dish字段修正               | **不处理**                        | 见下方"dish处理策略"         |
| **P3（跳过）** | 错别字检查                 | **不处理**                        | 工作量大，影响有限           |

### 1.3 dish字段处理策略（重要决策）

#### 决策：不在预处理阶段修正dish，而是在检索时让查询向量同时匹配name和dish

**原因分析**：

1. **成本过高**：用LLM批量处理31万条dish=Unknown的记录
   - API成本：约32元
   - 时间成本：约8.6小时
   - 效果提升：仅5-10%

2. **检索时name权重足够高**：
   - 当前embedding策略：`name为主（60-70%）+ dish为辅（15-20%）+ keywords补充（10-15%）`
   - 即使dish错误或缺失，name的强信号足以保证检索质量

3. **实际检索场景验证**：

   ```
   用户查询: "宫保鸡丁怎么做"

   食谱A: name="宫保鸡丁", dish="川菜"
     → Embedding: "宫保鸡丁的做法 川菜"
     → 相似度: 0.89 ✅ 最高匹配

   食谱B: name="宫保鸡丁", dish="Unknown"
     → Embedding: "宫保鸡丁的做法"
     → 相似度: 0.82 ✅ 仍能准确匹配

   食谱C: name="麻婆豆腐", dish="川菜"
     → Embedding: "麻婆豆腐的做法 川菜"
     → 相似度: 0.35 ❌ 自然排除
   ```

**结论**：**保持dish原样，依赖检索时name+dish同时匹配** —— 零成本达到同等效果

### 1.4 实现模块

**文件位置**：`src/data/cleaner.py`

**核心功能**：

- `remove_duplicates()` - MD5哈希去重
- `remove_empty_instructions()` - 删除空步骤
- `fill_empty_fields()` - 填充空字段
- `analyze_data_quality()` - 数据质量分析

**使用方式**：

```python

clean_recipe_data(
    input_file=Path("data/recipe_corpus_full.json"),
    output_file=Path("data/processed/recipes_cleaned.json"),
    limit=10000,  # 先测试1万条
    enable_quality_filter=True,

```

## 二、营养数据关联方案

### 2.1 数据源

**主营养数据源**：中国食物成分表（Excel）

- 本地离线查询
- 覆盖常见食材
- 数据权威准确

**补充营养数据源**：FatSecret API

- 仅在本地缺失时调用
- 缓存API结果，避免重复调用
- 控制月度额度

### 2.2 关联流程（三阶段）

```
┌─────────────────────────────────────────────────────────┐
│ 阶段1: 食材解析（ingredient_parser.py）                  │
├─────────────────────────────────────────────────────────┤
│ 输入: "1kg羊肉" 或 "5片姜" 或 "适量花椒"                  │
│ 处理:                                                   │
│   1. 正则提取数量/单位/食材名                             │
│   2. 处理修饰词（"大片"、"小根"等）                       │
│   3. 标准化食材名（同义词处理）                           │
│ 输出: {"name": "羊肉", "quantity": 1, "unit": "kg"}    │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│ 阶段2: 营养数据查询（nutrition_loader.py）               │
├─────────────────────────────────────────────────────────┤
│ 输入: "羊肉"                                             │
│ 查询流程:                                               │
│   1. 本地Excel查询（O(1)哈希表）                         │
│   2. 同义词匹配（模糊匹配）                              │
│   3. FatSecret API补全（缓存结果）                       │
│ 输出: {"energy": 203, "protein": 19, "fat": 12, ...}  │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│ 阶段3: 食谱营养聚合（nutrition_aggregator.py）           │
├─────────────────────────────────────────────────────────┤
│ 输入: {"recipeIngredient": ["1kg羊肉", "5片姜", ...]}  │
│ 处理:                                                   │
│   1. 逐个解析食材                                       │
│   2. 查询各自营养                                       │
│   3. 按数量加权汇总                                     │
│ 输出: {"energy": 2500, "protein": 150, "fat": 80}     │
└─────────────────────────────────────────────────────────┘
```

### 2.3 关联键值

**使用食材标准名作为关联键**

**挑战**：

- "羊肉" vs "羔羊肉" vs "绵羊肉" → 同义词处理
- "西红柿" vs "番茄" → 别名映射
- "1kg羊肉" → 提取"羊肉" → 查询营养

**解决方案**：

```python
# 构建同义词词典
INGREDIENT_SYNONYMS = {
    "羊肉": ["羔羊肉", "绵羊肉", "山羊肉"],
    "番茄": ["西红柿", "洋柿子"],
    "土豆": ["马铃薯", "洋芋"],
    # ...
}

# 查询时先标准化
def normalize_ingredient(name: str) -> str:
    for standard, aliases in INGREDIENT_SYNONYMS.items():
        if name in aliases or name == standard:
            return standard
    return name
```

### 2.4 实现模块（待开发）

**模块1**：`src/data/ingredient_parser.py`

- 解析食材字符串（数量+单位+名称）
- 处理特殊格式（"适量"、"少许"等）
- 同义词标准化

**模块2**：`src/data/nutrition_loader.py`

- 加载Excel到内存/SQLite
- 构建食材名→营养成分映射（哈希表）
- FatSecret API集成与缓存

**模块3**：`src/pipeline/nutrition_aggregator.py`

- 食材营养聚合为食谱营养
- 支持营养约束过滤（低脂/高蛋白）
- 食材禁忌检查

### 2.5 与RAG Pipeline的集成

```python
# 在检索时应用营养约束
query = "低脂高蛋白的健身餐"
constraints = {
    "max_fat": 30,      # 每份脂肪<30g
    "min_protein": 25,  # 每份蛋白质>25g
}

# 检索候选食谱
candidates = rag_pipeline.retrieve(query, top_k=50)

# 计算营养并过滤
filtered = []
for recipe in candidates:
    nutrition = nutrition_aggregator.aggregate_recipe(recipe)
    if nutrition['fat'] <= 30 and nutrition['protein'] >= 25:
        filtered.append((recipe, nutrition))

# 返回结果（附带营养信息）
return filtered[:10]
```

## 三、数据集规模决策

### 3.1 问题分析

**原始数据量**：155万条食谱

**问题**：

- ✅ 优势：体现数据规模，展示工程能力
- ❌ 劣势：索引构建耗时（10-20小时），查询可能较慢，演示时加载慢

### 3.2 分层数据集策略

| 数据集类型        | 数据量            | 用途                   | 构建时间（估算） | 说明                   |
| ----------------- | ----------------- | ---------------------- | ---------------- | ---------------------- |
| **快速验证集**    | 5,000-10,000      | 开发调试、功能测试     | 5-10分钟         | 快速迭代，验证代码逻辑 |
| **演示数据集** ⭐ | **30,000-50,000** | **阶段检查、答辩演示** | **30-60分钟**    | **推荐用于毕设**       |
| **完整数据集**    | 100,000-300,000   | 论文数据支撑、性能测试 | 2-5小时          | 体现工程能力           |
| **可选扩展**      | 全量155万         | "可扩展性"论述点       | 10-20小时        | 可选，不必须           |

### 3.3 最终决策：**5万条精选食谱作为演示数据集**

**理由**：

1. **兼顾质量与规模**：足够大以展示系统能力，又不至于影响性能
2. **构建时间可控**：30-60分钟可接受
3. **演示体验良好**：查询响应快（<2秒）
4. **毕设答辩适用**：导师能直观感受系统效果

### 3.4 筛选标准

**从155万条中筛选5万条的标准**：

```python
# 第一步：必须条件（估计剩余100-120万）
valid_recipes = [r for r in recipes if
    len(r['recipeInstructions']) >= 3 and    # 步骤完整
    len(r['recipeIngredient']) >= 3 and      # 食材丰富
    r['name']                                # 有菜名
]

# 第二步：质量排序
ranked_recipes = sorted(valid_recipes, key=lambda r: (
    0 if r['dish'] == 'Unknown' else 1,      # dish已知优先
    len(r['keywords']),                       # keywords多优先
    0 if not r['description'] else 1,         # 有描述优先
    len(r['recipeInstructions']),             # 步骤详细优先
), reverse=True)

# 第三步：分类采样（确保各菜系均衡）
# 避免某一菜系占比过高（如"家常菜"）
final_recipes = stratified_sample(ranked_recipes, n=50000)
```

### 3.5 论文中的表述

**不要说**："只用了5万条数据"（显得数据少）

**应该说**：

> "从155万条原始食谱数据中，经过去重、空值过滤、质量评估等数据清洗步骤，并按照步骤完整度、食材丰富度、描述详细度等指标进行质量排序，最终筛选出5万条高质量食谱构建知识库。这一规模既能充分验证RAG系统的检索与生成能力，又能保证查询响应速度满足实际使用需求。"

**体现的能力**：

- ✅ 数据清洗能力
- ✅ 质量评估能力
- ✅ 工程权衡能力（性能 vs 规模）

## 总结

本文档记录了食谱RAG系统的五个核心决策：

1. **数据清洗**：基础清洗为主，不修正dish字段，依赖检索时name+dish同时匹配
2. **营养关联**：三阶段流程（解析→查询→聚合），本地Excel为主，API补全为辅
3. **数据集规模**：5万条精选食谱，兼顾质量与性能
4. **硬件分工**：旧电脑开发准备，新电脑构建测试，Git同步代码
5. **20天计划**：三阶段（数据准备7天 + 系统完善7天 + 整合论文6天）

这些决策基于**工程务实**的原则，在有限时间内达到毕设要求的最佳方案。
